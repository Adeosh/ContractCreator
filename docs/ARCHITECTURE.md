# Архитектура проекта ContractCreator

Данный документ содержит детальное описание архитектурных принципов, структуры слоев и ключевых технических решений, реализованных в ERP-системе ContractCreator.

Проект спроектирован с упором на масштабируемость и тестируемость на основе **Clean Architecture (Чистой архитектуры)** и элементов **Domain-Driven Design (DDD)**. Взаимодействие слоев строго регламентировано: зависимости направлены от внешних слоев (инфраструктура, интерфейс) к внутренним (ядро бизнес-логики).

---

## 1. 🏗 Структура решения (Слои)

Решение разделено на 5 основных проектов, каждый из которых имеет строгую зону ответственности. Слой `Domain` является полностью независимым, тогда как остальные слои строятся вокруг него.

### 1.1 ContractCreator.Domain (Ядро)

Слой инкапсулирует фундаментальную бизнес-логику и правила предметной области. Не имеет зависимостей от библиотек доступа к данным или фреймворков.

* **Entities (Сущности):** Классы с уникальной идентификацией, отражающие бизнес-объекты (`Firm`, `Contract`, `Counterparty`, `Worker`). Содержат логику изменения собственного состояния.
* **Value Objects (Объекты-значения):** Неизменяемые (Immutable) типы, описывающие характеристики без собственной идентичности (например, `EmailAddress`, `AddressData`). Гарантируют свою валидность в момент создания.
* **Абстракции инфраструктуры:** Интерфейсы репозиториев (`IRepository<T>`), Unit of Work (`IUnitOfWork`) и абстрактные классы спецификаций (`ISpecification<T>`).

### 1.2 ContractCreator.Application (Слой сценариев)

Отвечает за оркестрацию бизнес-сценариев (Use Cases).

* **Сервисы (`I***Service`):** Реализуют конкретные операции (например, подписание контракта, создание накладной). Сервисы обращаются к репозиториям, извлекают сущности, применяют к ним бизнес-правила и сохраняют результат.
* **Маппинг:** Отвечает за преобразование доменных сущностей (Entities) в объекты передачи данных (DTO) для их безопасной передачи за пределы ядра.

### 3.3 ContractCreator.Infrastructure (Инфраструктура)

Реализует технические детали взаимодействия с внешним миром.

* **Доступ к данным:** Реализация интерфейсов `IRepository<T>` и `IUnitOfWork` с использованием Entity Framework Core (`AppDbContext`). Включает в себя миграции и Fluent-конфигурации сущностей БД.
* **Провайдеры:** Службы для работы с файловой системой (сохранение сканов, печатей), генерации файлов и взаимодействия с внешними API.

### 1.4 ContractCreator.Shared (Общие контракты)

Легковесный проект-контракт, обеспечивающий развязку между ядром и клиентским приложением.

* **DTO (Data Transfer Objects):** Плоские структуры данных без логики. Исключают попадание EF Core-сущностей в слой UI (решает проблемы ленивой загрузки и отслеживания изменений вне контекста БД).
* **Enums и Exceptions:** Перечисления (`LegalFormType`, `TaxationSystemType`) и общие структуры исключений.

### 1.5 ContractCreator.UI (Представление)

Клиентское приложение. Отвечает исключительно за отображение данных (Views, ViewModels) и делегирование пользовательского ввода слою Application. Ничего не знает о `DbContext` или доменных сущностях.

## 2. 🔄 Поток управления и данных (Data Flow)

Процесс обработки любого запроса строго следует правилу пересечения границ (Boundary Crossing). Рассмотрим пример загрузки и сохранения контрагента:

1. **Инициация:** Слой UI запрашивает данные у `CounterpartyService` (Слой Application).
2. **Извлечение:** `CounterpartyService` вызывает метод репозитория (в слое Infrastructure), передавая ему критерии выборки.
3. **Материализация:** EF Core выполняет SQL-запрос, поднимает Entity из базы данных и возвращает в Application.
4. **Маппинг:** Application с помощью Mapster преобразует тяжелую Entity в плоский `CounterpartyDto`.
5. **Отображение:** Слой UI отображает DTO пользователю.
6. **Модификация и Сохранение:** Измененный DTO отправляется обратно в `CounterpartyService`. Сервис извлекает оригинальную Entity, применяет изменения (с учетом бизнес-правил Domain), и вызывает `CommitTransactionAsync` у `UnitOfWork`.

## 3. 💾 Доступ к данным: Repository, Unit of Work и Спецификации

Архитектура доступа к данным спроектирована для максимальной тестируемости и минимизации дублирования LINQ-запросов.

### 3.1 Обобщенный Репозиторий и IUnitOfWork

* В проекте используется единый обобщенный класс `Repository<T>`. Специфичные репозитории создаются крайне редко, что предотвращает "раздувание" инфраструктурного слоя.
* **Управление транзакциями:** Класс `UnitOfWork` контролирует жизненный цикл `DbContext`. Он использует ленивую инициализацию: экземпляры репозиториев создаются через `Activator.CreateInstance` и кэшируются в `Hashtable`. Методы `BeginTransactionAsync`, `CommitTransactionAsync` и `RollbackTransactionAsync` гарантируют целостность данных при сложных операциях (например, сохранении договора вместе с десятками позиций спецификации).

### 3.2 Паттерн Спецификация (Specification Pattern)

Для решения проблемы размазывания фильтрации (`Where`), жадной загрузки (`Include`) и сортировки по всему слою Application внедрен паттерн Спецификация.

* **`BaseSpecification<T>`:** Класс в доменном слое, инкапсулирующий логику формирования выборки. Спецификации можно тестировать с помощью обычных Unit-тестов.
* **`SpecificationEvaluator<T>`:** Инфраструктурный компонент, который берет абстрактную спецификацию и транслирует ее в цепочку методов `IQueryable` для Entity Framework Core. Автоматически обрабатывает:
* Критерии фильтрации (Criteria).
* Включения связанных сущностей (`Include`, `IncludeStrings`).
* Сортировку (`OrderBy`, `OrderByDescending`).
* Пагинацию (`Skip`, `Take`).

## 4. 🔄 Маппинг и Изоляция Домена (Mapster)

Используется высокопроизводительная библиотека **Mapster**. Настройка конфигурации изолирована в классе `MappingConfig` в слое Application.

* **Работа с Value Objects:** Инфраструктура маппинга берет на себя ответственность за "разворачивание" сложных доменных объектов при передаче на клиент и их обратную "сборку" при сохранении.
* *Пример:* Доменное свойство `EmailAddress` имеет внутреннюю валидацию. Mapster настроен так, чтобы в `DTO` уходила просто строка, а при маппинге `DTO -> Entity` автоматически вызывался конструктор `new EmailAddress(str)`:
```csharp
config.NewConfig<EmailAddress, string>().MapWith(email => email.Value);
config.NewConfig<string, EmailAddress>().MapWith(str => string.IsNullOrWhiteSpace(str) ? null! : new EmailAddress(str));

```

* **Сохранение ссылок:** Включена опция `PreserveReference(true)` для предотвращения зацикливания при маппинге сложных графов (например, Договор <-> Спецификации <-> Накладные).

## 5. ⚡ Оптимизация работы с Enum

Система глубоко интегрирована с перечислениями. Стандартные вызовы `.ToString()` и рефлексивный поиск атрибутов `[Description]` создают высокую нагрузку на Garbage Collector и CPU при рендеринге больших таблиц.

* **`EnumsExtension`:** Статический класс-хелпер, использующий `ConcurrentDictionary` для кэширования метаданных перечислений.
* При первом запросе происходит чтение атрибута (`DescriptionAttribute` или кастомного `CustomAttribute`) через `FieldInfo.GetCustomAttribute`. Результат помещается в потокобезопасный кэш. Все последующие запросы к этому значению выполняются за время `O(1)`, полностью минуя рефлексию.

## 6. 🛡 Обработка ошибок и фоновые задачи

В приложении выстроена централизованная система перехвата исключений, разделяющая ошибки бизнеса и системные сбои.

### 6.1 Исключения домена (UserMessageException)

Если операция нарушает бизнес-логику (попытка удалить активную фирму, некорректная сумма спецификации), выбрасывается `UserMessageException`. Этот тип исключений имеет уровни (Info, Warning, Error) и предназначен исключительно для информирования пользователя.

### 6.2 Управление асинхронностью (SafeFireAndForget)

Поскольку многие команды UI имеют сигнатуру `async void` (Fire-and-forget), необработанные исключения в них могут привести к жесткому падению приложения.

* Имплементирован метод расширения `SafeFireAndForget()`. Он оборачивает выполнение `Task` в блок `try/catch`.
* Если перехватывается `UserMessageException` — ошибка выводится в UI через `IUserDialogService`.
* Если происходит системный сбой (например, обрыв сети или `SqlException`) — ошибка скрытно логируется через **Serilog**, а процесс не прерывается фатально.

## 7. 🖨 Архитектура печатных форм

Процесс генерации документов (договоров, счетов, актов) изолирован в отдельный пайплайн, чтобы не перегружать сервисы транзакционной бизнес-логики.

1. **Агрегация данных:** При запросе на печать, Application-слой выполняет слияние графа объектов. Извлекаются реквизиты собственной фирмы, контрагента, ответственных подписантов, банковских счетов и строк спецификации.
2. **Трансформация в Print-DTO:** Вся собранная структура конвертируется в специальные плоские модели из пространства имен `Shared/DTOs/PrintForms`. В этих моделях уже произведены нужные математические округления сумм и расчетов НДС.
3. **Генерация:** Независимый сервис генерации принимает DTO-модель и подставляет переменные в шаблон. Архитектура позволяет легко менять движок генерации (HTML-шаблонизация, прямая запись в PDF или подстановка закладок в Word/Excel), не затрагивая при этом ни слой извлечения данных, ни пользовательский интерфейс.

## 8. 🧭 Навигация и управление состоянием экранов

Несмотря на то, что это относится к пограничному слою (между UI и Application), механизм навигации реализован на основе строгих архитектурных шаблонов для контроля жизненного цикла.

* **Изоляция истории:** `NavigationService` инкапсулирует `Stack<ViewModelBase>`. Метод `MapsBack` гарантирует возврат к предыдущему экрану с сохранением 100% его состояния в памяти (значения фильтров, открытые узлы деревьев).
* **Управление памятью (`INavigatedAware`):** Интерфейс обязывает ViewModels реализовывать методы `OnNavigatedToAsync` и `OnNavigatedFromAsync`. Это фундаментально важно для отписки от глобальных событий (Event Aggregator) при уходе со страницы, что предотвращает классические утечки памяти (Memory Leaks).
* **Контекст (EditorParams):** Передача состояния между экранами стандартизирована через `EditorParams`. Для оптимизации работы со списками "на лету" без обращения к БД реализован `InMemoryEditorParams<T>`, передающий прямую ссылку на коллекцию в памяти.
